# FIFA World Cup Stadium Operations — Backend

Real-time (simulated) stadium telemetry backend with a Gemini-powered **FIFA AI Operations
Commander** for risk assessment and operational recommendations.

## Stack

- Node.js + Express
- `@google/genai` (Gemini API)
- dotenv, cors, helmet, morgan

## Project structure

```
backend/
├── src/
│   ├── config/        # env + gemini client setup
│   ├── controllers/    # request handlers
│   ├── routes/         # express routers
│   ├── services/       # simulation engine, state store, gemini service
│   ├── middleware/      # error handling, logging, 404
│   ├── prompts/         # Gemini prompt builder
│   ├── data/            # stadium reference data + mock generators
│   ├── utils/            # logger, async handler, random helpers
│   └── index.js          # app entrypoint
├── package.json
├── .env.example
└── README.md
```

## Setup

```bash
cd backend
npm install
cp .env.example .env
# edit .env and set GEMINI_API_KEY (optional — a deterministic offline
# fallback is used automatically if it's not set)
npm start
```

The server starts on `http://localhost:8080` by default (see `PORT` in `.env`).

## Endpoints

### `GET /api/health`
Service health, uptime, simulation tick count, whether Gemini is configured.

### `GET /api/dashboard`
Full live snapshot of all simulated stadiums (CCTV, crowd density, ticket scans, weather,
parking, metro, emergency, broadcast, volunteers, IoT), aggregate stats, and recent events.
Pass `?stadiumId=stad-001` to scope to a single venue.

### `GET /api/events`
Recent operational events (auto-generated by the simulation + manually posted ones).
Supports `?limit=50`.

### `POST /api/events`
Manually log an operational event.

```json
{
  "stadiumId": "stad-001",
  "type": "GATE_CONGESTION",
  "message": "Gate 4 queue exceeding capacity.",
  "severity": "warning"
}
```

### `POST /api/reason`
Triggers a Gemini reasoning pass acting as the **FIFA AI Operations Commander**. Returns
structured JSON with risk level, confidence, summary, predictions, recommendations, volunteer
deployment plan, broadcast decision, and announcements.

```json
{
  "stadiumId": "stad-001",   // optional — omit to reason over all stadiums
  "focus": "Assess crowd safety risk at the north gates"  // optional
}
```

If `GEMINI_API_KEY` is not configured, or the live call fails for any reason, this endpoint
transparently falls back to a deterministic local assessment so the response contract never
breaks for the frontend (`reasoning.meta.source` will read `"fallback"` vs `"gemini"`).

## Simulation engine

On startup, `src/services/simulation.service.js` seeds every stadium with a realistic snapshot
and then runs a `setInterval` loop (`SIM_TICK_INTERVAL_MS`, default 4000ms) that smoothly drifts
every numeric metric and occasionally flips categorical ones (weather, parking, metro, broadcast
status), so telemetry evolves organically instead of jumping randomly. It also probabilistically
emits operational events (crowd density spikes, emergencies, parking/metro disruptions, broadcast
issues, IoT anomalies) into the rolling event log consumed by `/api/events` and `/api/dashboard`.
